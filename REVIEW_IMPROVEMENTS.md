# Код-ревью URSASS Backend и предложения по улучшению

## Что проверено
- Архитектура API (`server.js`, `routes/*`), модели (`models/*`), ограничения запросов и базовые античит-проверки.
- Консистентность валидации подписи/времени между endpoint'ами.
- Наблюдаемость и безопасность (rate limit, security event logging, replay-protection).

## Сильные стороны
1. Есть верификация EIP-191 подписи в критичных write-операциях.
2. Есть базовая античит-логика и журналирование подозрительных событий (`SecurityEvent`).
3. Реализованы метрики и middleware для мониторинга запросов.
4. Учтена обратная совместимость API (`/api/v1/*`, alias для upgrade keys).

## Улучшения (приоритет P0–P2)

### P0 (высокий приоритет) // выполнено

#### 1) Добавить rate limiter на `/api/account/link/verify-telegram`
**Проблема:** endpoint изменения состояния аккаунта не ограничен по частоте, что открывает путь к brute-force кода или DoS на связку аккаунтов.

**Где видно:** в `routes/account.js` для `POST /link/verify-telegram` нет `writeLimiter`, в отличие от других write-route.

**Рекомендация:** добавить `writeLimiter` (или отдельный более строгий limiter + per-IP/per-telegramId ключ).

---

#### 2) Унифицировать проверку свежести timestamp для wallet auth/link // выполнено
**Проблема:** подпись проверяется, но сам timestamp в `auth/wallet` и `link/wallet` не валидируется по окну времени. Это делает возможным replay старой валидной подписи.

**Где видно:** `routes/account.js` в `/auth/wallet` и `/link/wallet` timestamp участвует в сообщении, но нет проверки `Date.now() - timestamp`.

**Рекомендация:** внедрить общее правило (например, ±5–10 минут), как уже сделано в других write-endpoint'ах.

---

#### 3) Усилить дедупликацию результатов игры на уровне БД // выполнено
**Проблема:** в `/api/leaderboard/save` дубль проверяется через `findOne({ signature })`, но поле `signature` в модели не уникально. При race-condition два параллельных запроса могут пройти проверку до insert.

**Где видно:** `routes/leaderboard.js` (проверка перед сохранением) и `models/GameResult.js` (нет `unique` для `signature`).

**Рекомендация:**
- добавить уникальный индекс по `signature`,
- обрабатывать Mongo duplicate key error и возвращать корректный 409/400.

---

### P1 (средний приоритет)

#### 4) Сделать обновление Player + GameResult атомарным // выполнено
**Проблема:** сохранение результата и обновление агрегатов игрока выполняются несколькими операциями. При ошибке между шагами возможна частичная запись и рассинхронизация.

**Рекомендация:** использовать транзакцию MongoDB (`session.withTransaction`) для `GameResult` + `Player`.

---

#### 5) Перевести `console.log/error` на единый structured logger // выполнено
**Проблема:** в коде смешаны `console.*` и `logger.*`, из-за чего формат логов в проде становится неоднородным.

**Рекомендация:** заменить `console.*` на `logger.info/warn/error` во всех route-файлах и utils.

---

#### 6) Логировать попытку покупки до бизнес-логики // выполнено
**Проблема:** в `/api/store/buy` событие `purchase_attempt` пишется после успешной покупки. Это ослабляет анализ подозрительной активности (failed attempts не видны).

**Рекомендация:** писать два типа событий:
- `purchase_attempt` — в начале обработки,
- `purchase_result` (`success`/`fail`, reason) — в конце.

---

### P2 (желательно)

#### 7) Вынести общие security-утилиты 
**Проблема:** `logSecurityEvent`, timestamp validation и части anti-cheat логики дублируются между route-файлами.

**Рекомендация:**
- вынести в `utils/security.js`:
  - `validateTimestampWindow(...)`
  - `logSecurityEvent(...)`
  - `normalizeWallet(...)`
- покрыть это unit-тестами.

---

#### 8) Добавить минимальный набор автотестов API // выполнено
**Проблема:** отсутствуют интеграционные тесты для критичных сценариев (подписи, replay, ride consume).

**Рекомендация:**
- smoke/integration на:
  - `POST /api/leaderboard/save` (валидная/невалидная подпись, replay),
  - `POST /api/store/buy` (insufficient funds, tier mismatch),
  - `POST /api/account/link/verify-telegram` (rate limit + expired code).

---

## Быстрый план внедрения
1. Закрыть P0 (limiter + timestamp window + unique index signature).
2. Добавить обработку duplicate key ошибок и единый код ответа.
3. Перейти на транзакции для сохранения результата игры и статистики игрока.
4. Рефакторинг логирования и вынос общих security-утилит.
5. Добавить интеграционные тесты на критичные endpoint'ы.

## Ожидаемый эффект
- Снижение риска brute-force/replay атак.
- Меньше дублей и гонок при сохранении результатов.
- Более чистая наблюдаемость (логи + метрики + security events).
- Выше предсказуемость при релизах за счет тестов.
